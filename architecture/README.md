# Архитектура email-аутрича (1200 адресов)

Архитектура e-mail аутрич-платформы на территории РФ (без AWS), рассчитанной на 1 200 адресов, с упором на отказоустойчивость и бюджет.

## Общая схема

- **Источник контактов** (CRM/BI) → очередь отправки → контроллер рассылки → пул MTA-сервисов → Интернет
- **Логирование и мониторинг** → хранилище логов → дэшборды и алерты
- **Обработка bounce/complaints** → обратная очередь → корректировка репутации

## Компоненты и сервисы

- **Облачная платформа:** Yandex Cloud (альтернатива VK Cloud Solutions или Selectel)
- **Compute:** Cloud VM (Managed Instance Group) или Yandex Managed Service for Kubernetes
- **Очередь задач:** RabbitMQ на VM (или Yandex Message Queue)
- **DNS и IP:** Yandex.DNS + Floating IP (для ротации исходящих)
- **Хранение логов:** Yandex Object Storage
- **Мониторинг:** Yandex Monitoring + Grafana/Prometheus
- **CI/CD и проверка DKIM/SPF/DMARC:** GitLab CI, Ansible

## Мультитенантность (клиенты и направления)

- **Раздельные очереди или теги по клиентам:** каждому клиенту назначается отдельная очередь RabbitMQ (vhost) или тег в сообщениях; контроллер маршрутизирует задачи по `client_id`. Изоляция гарантирует, что сбой одного клиента не блокирует остальных.
- **Лимиты по направлениям:** контроллер рассылки хранит rate-limit на каждое направление (campaign_type); лимит задаётся в конфиге (писем/час, писем/день). Превышение — задача возвращается в очередь с отложенной доставкой (TTL/delay).
- **Изоляция метрик и логов:** метрики Prometheus помечаются лейблами `client_id`, `direction`; логи в Object Storage пишутся в префиксы `logs/{client_id}/{direction}/`. В Grafana — отдельные дашборды или фильтры по клиенту и направлению.

## Ротация и балансировка нагрузки

- Разбить 1 200 адресов по 4–5 поддоменам (mail1.xxx…mail5.xxx)
- На каждый поддомен закрепить по 3–4 Floating IP
- В DNS настроить round-robin A-записи поддоменов на все IP
- Контроллер рассылки берёт задачу из очереди и отсылает через случайно выбранный VM/контейнер
- Масштабирование VM-группы по CPU/длине очереди (autoscaler)

## Мониторинг и авто-ремедиация

- Сбор метрик RabbitMQ, MTA-контейнеров, bounce/complaint rate
- Алёрты в Slack/Telegram при росте отказов, падении deliverability, упаде VM
- Скрипт-проверка репутации IP (SenderScore, Talos) → если < threshold, «холодный» IP вводится в ротацию, «горячий» — вычеркивается

## Риски и смягчение

- **Блокировка IP:** держать резервный пул Floating IP; оперативная подмена в DNS
- **Потеря очереди:** репликация RabbitMQ (кластер в 3 AZ) или переход на Yandex Message Queue с DLQ
- **Перегрузка кластера:** autoscaling + лимит скорости на контроллере
- **Неправильная конфигурация SPF/DKIM/DMARC:** pre-deploy тесты в CI, авто-проверка DNS

## Примерная стоимость (Yandex Cloud, ₽/мес)

| Компонент | Сумма |
|-----------|-------|
| Compute (4 × s1.small, 2 vCPU+2 GB) | 4 400 ₽ |
| RabbitMQ (2 × s1.micro) | 1 200 ₽ |
| Floating IP (20 IP × 100) | 2 000 ₽ |
| Object Storage и трафик | 500 ₽ |
| Yandex Monitoring + DNS | 300 ₽ |
| **Итого** | **≈ 8 400 ₽ (~$110)** |

При необходимости интеграции в VK Cloud или Selectel цифры сопоставимы. Автономный on-premise разворот (VMware/KVM) потребует примерно тех же VM-ресурсов плюс свои расходы на софт-лицензии.
